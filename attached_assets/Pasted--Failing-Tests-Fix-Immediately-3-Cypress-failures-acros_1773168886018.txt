🟠 Architecture & Data Fetching
staleTime: Infinity is too aggressive globally
In queryClient.ts, all queries are set to never expire. This means the UI won't reflect server changes (new workouts from Strava sync, plan updates) until the user manually refreshes. Consider tiered stale times:
tsstaleTime: 5 * 60 * 1000, // 5 min default
// Override per-query for highly dynamic data like timeline
getQueryFn URL construction is fragile
queryKey.join("/") will break if any key segment contains a /, or if array-style query keys with objects are ever used. Better to enforce that queryKeys are always full paths.
retry: false on all queries
Transient network blips will surface as errors to users. At minimum, allow 1 retry for GET queries.

🟠 Security
Cron secret exposed via query parameter
In server/routes/email.ts, the cron secret can be passed as ?secret=... in the URL, which risks it showing up in server logs and browser history. Accept it via header only:
tsconst secret = req.headers["x-cron-secret"] as string;
// Remove: || (req.query.secret as string)
req: any throughout route handlers
Every authenticated route casts req to any to access req.user. This bypasses TypeScript's safety net. Define a typed AuthenticatedRequest interface extending Express's Request so getUserId() is fully type-safe.

🟡 Code Quality
useOnboarding uses localStorage without abstraction
localStorage.getItem("hyrox-onboarding-complete") is called directly but never set in the same hook — the setter must be somewhere else. This is easy to lose track of. Centralise with a small useLocalStorage hook or a constant key name.
setTimeout in useOnboarding for coach panel
tssetTimeout(() => {
  if (!isCurrentlyMobile) setCoachOpen(true);
}, 500);
Magic 500ms delays are fragile (race conditions, layout shifts). Prefer a callback after onboarding animation completes, or use requestAnimationFrame.
Preview AI model in production
gemini-3-flash-preview is a preview model. Preview models can be deprecated without notice. Plan a migration path to a stable release channel.
console.error instead of structured logging
Errors are logged with console.error throughout. Since Sentry is already integrated, errors should be captured with Sentry.captureException() for server-side errors, not just logged to stdout where they may be missed.
